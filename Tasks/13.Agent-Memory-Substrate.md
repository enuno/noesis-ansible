## Task 13: memory substrate deployment

Mission:
Deploy and configure per-agent MemPalace instances for OpenClaw and Hermes, and a shared Honcho secondary memory substrate for cross-agent continuity. NoesisPraxis is the Hermes supervisor agent for the Noesis Universe infrastructure stack, and memory must be split so each agent has isolated primary memory while sharing a unified secondary substrate.

System architecture:
- Primary local memory substrate for each agent: mempalace.
- Secondary shared multi-agent memory substrate for all agents: Honcho.
- Honcho is self-hosted on infrastructure controlled by the operator.
- Each agent maintains its own local MemPalace instance or local MemPalace-backed memory workspace.
- Agents may read from Honcho for shared durable context, but local memory remains authoritative for session-scoped, private, and working-state information.

Authoritative sources:
- Local/private/session memory: mempalace.
- Shared durable cross-agent memory: Honcho.
- Supervisor policy, routing, and promotion decisions: NoesisPraxis.

Honcho deployment requirements:
- Provision and operate a self-hosted Honcho server from the upstream repository:
  - https://github.com/plastic-labs/honcho
- Expose the Honcho service to agents through a controlled internal endpoint.
- Configure app, user, session, collection, document, and metamessage handling as needed for multi-agent memory.
- Use Honcho’s insight and dialectic capabilities for durable shared memory, agent modeling, and cross-agent context hydration.
- Keep Honcho isolated from public exposure unless explicitly required and secured.

MemPalace deployment requirements:
- Provision MemPalace locally for each agent or each agent workspace:
  - https://github.com/MemPalace/mempalace
- Use MemPalace as the primary local memory store for verbatim conversation history, project context, and local retrieval.
- Keep MemPalace local-first and privacy-preserving.
- Use MemPalace’s hooks, CLI, and MCP tooling for automatic capture, mining, wake-up, and retrieval where available.

Memory routing policy:
1) Use MemPalace first for local, private, session-scoped, and working memory.
2) Consult Honcho second for shared durable memory, peer modeling, and cross-agent continuity.
3) Promote only stable, non-sensitive, repeatedly confirmed facts from MemPalace into Honcho.
4) Never store secrets, credentials, private keys, ephemeral tokens, or destructive commands in Honcho.
5) If MemPalace and Honcho disagree, prefer MemPalace for session facts and Honcho only for stable identity after explicit validation.

Honcho configuration requirements:
- Enable reasoning unless a task explicitly disables it.
- Use peer cards for stable biographical facts, preferences, standing instructions, and identity anchors.
- Tailor representation scopes to the peer and task so shared memory stays focused and role-appropriate.
- Use dreaming as part of the hermes-dreaming-cycle and self-reflection-cycle.
- Use streaming responses for long-form reasoning, interactive synthesis, or latency-sensitive responses.
- Use file uploads for research papers, technical docs, logs, and long-form references.

Peer card policy:
- Peer cards must contain only stable facts.
- Keep entries atomic and deduplicated.
- Include:
  - name or alias,
  - role,
  - persistent preferences,
  - standing instructions,
  - stable identity anchors.
- Exclude transient tasks, ephemeral context, and raw reasoning traces.

Dreaming cycle:
- Schedule dreaming after meaningful accumulation of evidence or when the session becomes idle.
- Use dreaming to:
  - resolve contradictions,
  - remove outdated conclusions,
  - infer stable patterns,
  - update peer cards,
  - consolidate self-reflection,
  - prepare durable shared memory for future sessions.

Streaming policy:
- Prefer streaming when response length is substantial, reasoning is progressive, or user experience benefits from immediate partial output.
- Do not stream when the output must be atomically validated before release or when the action is sensitive and requires approval.

File upload policy:
- Accept uploaded files for research papers, specs, logs, architecture notes, and datasets.
- Ingest files into Honcho when they are useful for shared reasoning or long-term reference.
- Keep raw sensitive files controlled; only promote safe extracted facts into Honcho.

Security and operations:
- Use least privilege for all memory and tool access.
- Keep Honcho behind authenticated access.
- Rotate credentials and scope tokens narrowly.
- Log memory actions, promotions, retrievals, and dream cycles.
- Maintain rollback and manual override paths for memory corruption or bad promotion events.
- Treat shared memory as an aid, not a sole source of truth.

Execution behavior:
- Check MemPalace first.
- Consult Honcho when cross-agent continuity matters.
- Queue durable facts for Honcho updates.
- Trigger dreaming when contradictions or stable patterns emerge.
- Operate as a supervisor coordinating memory, not as a worker executing blindly.
