Task:
Generate the inventory scaffolding for the NoesisPraxis Ansible repository so the repo can bootstrap end-to-end. Create the inventory directories and files exactly as specified below, with consistent host/group naming, sane defaults, and environment separation for local, Tailscale, and production use.

Repository inventory layout to generate exactly:

ansible/inventory/
├── local/
│   ├── hosts.ini
│   ├── group_vars/
│   │   └── all.yml
│   └── host_vars/
│       ├── localhost.yml
│       └── macbook-pro-m1pro.yml
├── tailscale/
│   ├── hosts.ini
│   ├── group_vars/
│   │   └── all.yml
│   └── host_vars/
│       └── macbook-pro-m1pro.yml

    ├── hosts.ini

General requirements:
- Create exactly the files above.
- Do not add extra inventory files in this task.
- Use clear host aliases and group names.
- Keep the inventories static and readable.
- Support both localhost and Tailscale connectivity.
- Keep production separate from local home-lab settings.
- Use YAML for vars files.
- Use INI for `hosts.ini` files unless there is a compelling reason otherwise.
- Keep all secrets out of these files; only reference vaulted or protected values.
- Include comments only where needed to clarify inventory intent.

Inventory behavior requirements:
- `inventory/local` is the default home-lab bootstrap inventory.
- `inventory/tailscale` is the remote management inventory used when Ansible connects over the tailnet to the Mac host.
- `inventory/production` is a separate environment for later scaled deployment and must not inherit local-only assumptions.
- `inventory/local` must include both a localhost target and the MacBook Pro target if needed for local testing.
- `inventory/tailscale` must define the MacBook Pro host alias with Tailscale-reachable connection variables.
- `inventory/production` must define a clean production host/group structure, even if minimal.
- Group variables should define environment-wide defaults for registry, runtime, networking, Telegram, backup, validation, and platform tuning.

Required content per file:

1. `inventory/local/hosts.ini`
- Define a local management host alias.
- Define a macOS host alias for the MacBook Pro M1 Pro.
- Include groups for:
  - `local`
  - `macos`
  - any agent-specific groups needed for initial bootstrap
- Set connection variables appropriately for local execution and optional SSH if used.

2. `inventory/local/group_vars/all.yml`

  - ACP

  - macOS agents

  - backup
  - validation

- Define localhost and tailnet bind settings if applicable.

3. `inventory/local/host_vars/localhost.yml`
- Define localhost-specific connection and role settings.
- Keep this minimal and safe.
- Use `ansible_connection: local` or equivalent.

4. `inventory/local/host_vars/macbook-pro-m1pro.yml`
- Define the MacBook Pro host alias.
- Include macOS-specific metadata and connection settings.
- Include resource-budget defaults for the M1 Pro / 16 GB host.
- Include localhost and Tailscale endpoint placeholders if needed.
- Keep secrets out of the file.

- Include any environment groups used for tailnet-managed services.

6. `inventory/tailscale/group_vars/all.yml`
- Define shared defaults for Tailscale-managed operations.

  - Ansible connection defaults for remote management

  - validation toggles

7. `inventory/tailscale/host_vars/macbook-pro-m1pro.yml`
- Define the host’s tailnet endpoint details.
- Include the Tailscale host alias and any transport variables.
- Include remote management considerations for NoesisPraxis communication over Tailscale.
- Keep the file concise and explicit.

8. `inventory/production/hosts.ini`
- Define a clean production inventory structure.
- Include at least one production group skeleton for future expansion.
- Keep production distinct from local and tailscale environments.
- Avoid local-only or Apple-specific assumptions.

9. `inventory/production/group_vars/all.yml`
- Define production-wide defaults with safe placeholders.
- Include toggles for the major stack components.
- Include stricter defaults for validation, security, and change control.
- Keep secrets externalized.
