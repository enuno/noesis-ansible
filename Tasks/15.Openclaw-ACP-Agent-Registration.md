Mission:
Create the Ansible playbook configuration portion for all OpenClaw agent playbooks, including generic OpenClaw deployments and specialized Noesis* OpenClaw agents, so that each agent exposes an ACP gateway for control via NoesisPraxis and is registered in the ACP, A2A, and Agent registries.

Primary goals:
- Generate the Ansible configuration layer for each OpenClaw agent playbook.
- Provision and configure an OpenClaw Gateway with ACP support.
- Expose the ACP gateway so NoesisPraxis can control and route work through it.
- Register each OpenClaw agent ACP gateway with:
  - ACP registry/discovery mechanisms,
  - A2A agent discovery and routing,
  - the relevant Agent registry / agent directory / identity layer used by the platform.
- Keep the solution consistent across generic deployments and specialized Noesis* agents.

Scope:
- Generic OpenClaw agent playbooks.
- Specialized Noesis* OpenClaw agent playbooks.
- Gateway configuration, ACP bridge configuration, registry metadata, and identity registration.
- Ansible variables, templates, roles, handlers, and inventory structure needed to support deployment.

OpenClaw ACP behavior:
- Treat `openclaw acp` as the Gateway-backed ACP bridge, not a standalone ACP-native runtime.
- Ensure the OpenClaw Gateway is running before ACP bridge startup.
- Configure the gateway target, session routing, and authentication as required.
- Validate that ACP sessions map correctly to gateway session keys.
- Use the gateway as the control plane for NoesisPraxis supervision.

Ansible implementation requirements:
- Create reusable Ansible roles for:
  - OpenClaw gateway installation and configuration,
  - ACP bridge configuration,
  - registry enrollment and metadata publishing,
  - service supervision and lifecycle management,
  - health checks and validation.
- Use group vars, host vars, and role defaults to parameterize:
  - gateway URL,
  - auth token or credential reference,
  - agent name,
  - agent type,
  - registry endpoints,
  - A2A identity metadata,
  - ACP bridge command and flags,
  - ports, sockets, and health checks.
- Keep secrets out of plaintext inventory and templates.
- Use Ansible Vault or an external secret manager for sensitive values.
- Ensure idempotency for all tasks.
- Add handlers for restart, reload, and registry re-registration when relevant config changes.

ACP gateway requirements:
- Configure the OpenClaw gateway endpoint to be reachable by NoesisPraxis.
- Expose ACP over stdio where needed for clients that use ACP bridge mode.
- Preserve session continuity and routing semantics.
- Support local and remote gateway targets.
- Add explicit validation that the gateway is healthy before registration or agent startup.

Registry requirements:
- Register each gateway and agent with the ACP registry layer.
- Publish A2A discovery metadata so peer agents can locate and delegate to it.
- Register the agent in the platform’s Agent registry / identity layer with:
  - canonical agent name,
  - role,
  - capabilities,
  - endpoint,
  - protocol support,
  - owner or supervisor,
  - trust or auth metadata as needed.
- Ensure registry records are updated on deploy, upgrade, and identity changes.
- Include cleanup or deregistration behavior for teardown.

Specialized Noesis* agent requirements:
- For each Noesis* agent variant, inherit the base OpenClaw ACP configuration.
- Override only role-specific capabilities, identity metadata, and registry fields.
- Keep the policy surface consistent across variants.
- Make the agent discoverable and controllable by NoesisPraxis through the ACP gateway.

Acceptance criteria:
- Every OpenClaw playbook has a generated Ansible config portion.
- Every agent exposes an ACP gateway usable by NoesisPraxis.
- Every agent is registered in ACP, A2A, and the relevant Agent registry.
- The Ansible output is idempotent, maintainable, and secret-safe.
- Generic and specialized Noesis* agents share a common role-based deployment pattern.
- The gateway is validated before registration and startup completes cleanly.
- The output includes documentation for deploy, upgrade, rollback, and deregistration.

Operational policy:
- Never hardcode secrets, credentials, or private keys.
- Fail fast if gateway validation or registry enrollment fails.
- Prefer explicit, auditable configuration over implicit discovery.
- Use least privilege for services and registry credentials.
- Keep control-plane endpoints isolated and authenticated.
- Treat registry writes as operationally sensitive and reversible.

Implementation output:
- Ansible roles and playbook fragments.
- Jinja2 templates for config files and service units.
- Inventory examples for generic and specialized agents.
- Registry enrollment tasks for ACP, A2A, and agent discovery.
- Validation tasks for gateway health and ACP bridge functionality.
- Documentation for install, update, rollback, and teardown.

Reference documentation:
- OpenClaw ACP docs: https://docs.openclaw.ai/cli/acp
- OpenClaw docs: https://docs.openclaw.ai
- OpenClaw ACP bridge behavior: https://docs.openclaw.ai/cli/acp
- A2A and agent interoperability should be treated as first-class discovery surfaces for cross-agent coordination.
