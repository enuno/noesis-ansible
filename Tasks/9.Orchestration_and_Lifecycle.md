Mission:
Use Ansible as the primary orchestration and lifecycle tool for the Noesis universe of back-end infrastructure, agent deployment, maintenance, updates, backups, and registry synchronization. The repository at ~/projects/noesis-ansible/ is the source of truth for all Noesis multi-agent infrastructure, including MCP servers and any associated services directly consumed by AI agents.

Operating model:
- Treat Ansible as the composable infrastructure control plane.
- Encode each capability as idempotent playbooks, roles, templates, inventories, and validation steps.
- Prefer small, reusable roles over monolithic scripts.
- Keep the repo organized so deployment, upgrade, backup, restore, and maintenance workflows can be run independently or as a chained stack.
- Preserve a local-first workflow so the repo can be developed and tested safely before any remote changes.

Source-of-truth requirements:
- ~/projects/noesis-ansible/ is the authoritative repo for:
  - agent runtimes
  - registries
  - MCP servers
  - service configs
  - backup and restore logic
  - validation and health checks
  - change tracking and rollout coordination
- Any service or agent that NoesisPraxis depends on should be represented here as code.
- Avoid manual configuration drift by rendering all runtime artifacts from Ansible variables and templates.

Secrets and vault policy:
- All secrets must originate in Bitwarden Secrets Manager and be fetched locally using the `bws` command.
- After fetching, secrets must be imported into Ansible Vault or Vault-managed local files so routine Ansible execution does not need live external Bitwarden calls.
- Never hardcode secrets in templates, inventories, playbooks, or role defaults.
- Any secret-bearing task must support local rotation, validation, and recovery.
- The repo must assume secrets can be refreshed from Bitwarden when necessary, but normal runs should consume Vault-managed local values.

Ansible architecture rules:
- Use inventories to separate local, tailnet/Tailscale, and production contexts.
- Use group_vars and host_vars to define environment-specific defaults.
- Use roles for reusable implementation units.
- Use templates for configs, registry entries, launchd plists, service definitions, and policy files.
- Use custom modules or filters only when a native Ansible primitive is not sufficient.
- Use handlers for restart/reload behavior.
- Use tags to support partial runs and phased execution.
- Use check mode and validation tasks to prove idempotency before change rollout.
- Use dynamic inventory where host discovery is better expressed as code than as static files.

Repository structure expectations:
- README.md explains the stack, layout, and execution flow.
- ansible.cfg sets repo-local defaults.
- requirements.yml declares the minimal dependency set.
- inventory/ contains local, tailscale, and production inventories.
- playbooks/ contains site.yml, master-stack.yml, and subsystem playbooks.
- roles/ contains one role per subsystem or platform concern.
- templates/ contains shared Jinja2 templates and policy/config artifacts.
- files/ contains schemas and utility scripts.
- scripts/ contains operational helpers for bootstrap, backup, restore, validation, and registry sync.

Stack coverage:
Implement and maintain the following as composable Ansible-managed subsystems:
- ACP registry
- ANS
- Agent Registry dashboard
- A2A registry
- MCPJungle MCP server registry
- Clawvisor security layer
- Telegram communications
- OpenClaw runtime
- Hermes runtime
- Noesis ClawDev for macOS
- Noesis HermesDev for macOS
- Tailscale networking
- backup, restore, validation, rollback, and synchronization workflows

Execution style:
- Break the stack into phases:
  1. Bootstrap
  2. Identity and registries
  3. Security and access
  4. Generic runtimes
  5. macOS runtimes
  6. Networking
  7. Communications
  8. Backup and restore
  9. Validation and rollback
- Keep `site.yml` as the operator entry point.
- Keep `master-stack.yml` as the phase orchestrator.
- Let each playbook include only the roles it needs.
- After each phase, run validation appropriate to that phase.
- Make rollback paths and maintenance paths explicit.
- Preserve idempotency at all times.

macOS and Tailscale requirements:
- The macOS deployment target is a MacBook Pro M1 Pro with 16 GB RAM.
- Use launchd for per-user service management where appropriate.
- Bind services to localhost and Tailscale as intended, not broadly to the public network.
- Keep CPU and RAM usage conservative.
- Ansible should be able to reach the Mac host over Tailscale.
- NoesisPraxis should communicate with the macOS agents over Tailscale for remote coordination.
- Validate both localhost and tailnet reachability.

MCP and service dependency rules:
- MCP servers and agent-consumed services must be defined as code and deployed by Ansible.
- Access to MCP servers should be mediated by approved registry entries and policy layers.
- Any service that agents depend on must have an install, configure, validate, backup, and restore path where applicable.
- Keep registry synchronization atomic and auditable.

Implementation guardrails:
- Prefer declarative tasks over shell commands.
- If a shell command is required, keep it short, safe, and idempotent.
- Use comments only where they clarify operational behavior.
- Use placeholders where real values are not yet known.
- Avoid Linux-only assumptions in macOS playbooks and roles.
- Avoid assumptions about network topology unless explicitly expressed in inventory or vars.
- Keep the repo safe to run in a local lab before productionizing.

Required output behavior:
- When given task files, execute them in order.
- Create or update repository files under ~/projects/noesis-ansible/ only.
- Do not skip dependencies silently.
- If a task cannot be completed, report the exact blocker and stop at the smallest safe boundary.
- Validate each subsystem before moving to the next task.
- Preserve a clean repository tree and valid YAML/JSON throughout.

Priority objective:
Build a coherent, composable Ansible library that can deploy, maintain, and validate the Noesis back-end stack end-to-end, with Vault-managed secrets sourced from Bitwarden via `bws`, and with the repo itself serving as the operational source of truth.
