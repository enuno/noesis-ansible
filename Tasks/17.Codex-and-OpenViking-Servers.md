Mission:
Create Ansible playbooks and associated configuration to install and configure three delegated development backends:
1) Codex Server
2) OpenViking
3) any Hermes/OpenClaw agent routing needed to delegate development tasks through those backends

This mission must produce two deployment versions for every playbook:
- Local installation version.
- k3s/k8s installation version, using Helm if available and appropriate.

Primary goals:
- Provision Codex Server in a controlled, repeatable, infrastructure-as-code manner.
- Provision OpenViking  as a controlled context-memory and retrieval backend.
- Configure Hermes, OpenClaw, and Noesis specialist agents to delegate development tasks to Codex when appropriate.
- Configure agents to use OpenViking as a context/memory backend where beneficial and approved.
- Keep local execution available as a fallback when the backend is unavailable or the task should not be delegated.
- Support both local and cluster-based operation for all components.

Scope:
- Codex Server installation and lifecycle management.
- OpenViking  installation and lifecycle management.
- Agent configuration for Hermes, OpenClaw, and Noesis* variants.
- Delegation policy for development tasks.
- Memory/context backend wiring.
- Registry, identity, and endpoint wiring if either backend is exposed as a service.
- Validation, rollback, and operational documentation.

Deployment version requirement:
For every service and every playbook, produce:
1) Local install version.
2) k3s/k8s install version.
3) Prefer Helm for k3s/k8s packaging when a Helm chart is available or practical.
4) If Helm is not available, provide clean Kubernetes manifests with equivalent behavior.

Codex deployment requirements:
- Install Codex using the supported local installation method for the target platform.
- Also provide a k8s/k3s deployment path if the workload is serviceable in-cluster.
- Manage authentication and provider configuration explicitly.
- Keep the runtime isolated, supervised, and restartable.

OpenViking  deployment requirements:
- Install OpenViking using the supported local installation method for the target platform.
- Also provide a k8s/k3s deployment path if the workload is serviceable in-cluster.
- Configure persistent storage, workspace/config directories, and service health checks.
- Expose only required ports.
- Ensure the service can be used as a shared context database / retrieval backend where approved.
- Keep configuration portable across local and cluster deployments.

Agent configuration requirements:
- Create reusable configuration for:
  - Hermes agents,
  - OpenClaw agents,
  - Noesis specialist agents,
  - subordinate coding or context workers.
- Set the routing policy so development tasks can be delegated to Codex.
- Set the context/memory policy so agents can use OpenViking where appropriate.
- Preserve direct local execution for sensitive, high-risk, or control-plane tasks.
- Support task routing for implementation, refactors, reviews, tests, and modular builds.

Routing and context policy:
- Delegate to Codex when the task is implementation-heavy, self-contained, or benefits from an external coding agent.
- Use OpenViking for durable context, retrieval, and context-file management where appropriate.
- Keep supervisor decisions with Hermes / NoesisPraxis.
- Do not delegate secrets, production credentials, or destructive infrastructure operations unless explicitly authorized.
- Require explicit approval for high-risk changes.
- Support fallback to local execution when Codex or OpenViking is unavailable.

Ansible implementation requirements:
- Create reusable roles for:
  - Codex install,
  - Codex auth/config,
  - OpenViking install,
  - OpenViking config/storage,
  - agent delegation policy,
  - agent context backend wiring,
  - service supervision,
  - health checks.
- Parameterize:
  - install method,
  - auth mode,
  - binary path,
  - service account,
  - internal endpoint,
  - storage path,
  - delegation thresholds,
  - logging and telemetry.
- Keep secrets out of plaintext inventory and templates.
- Use Ansible Vault or an external secret manager.
- Make all tasks idempotent.
- Include handlers for restart, reload, and reconfiguration.

Validation requirements:
- Verify Codex installs and starts successfully.
- Verify OpenViking installs and serves correctly.
- Verify Hermes can delegate coding tasks to Codex.
- Verify agents can query and use OpenViking where configured.
- Verify fallback behavior when either backend is unavailable.
- Confirm logs, errors, and execution traces are observable.

Operational requirements:
- Treat Codex as a delegated coding executor, not an unrestricted control plane.
- Treat OpenViking as a context backend, not a secret store.
- Use least privilege and clear approval gates.
- Separate routing policy from execution mechanics.
- Log task handoffs, retrieval calls, and completion status.
- Provide rollback and teardown procedures.

Deliverables:
1) Ansible role/playbook set for local Codex deployment.
2) Ansible role/playbook set for k3s/k8s Codex deployment.
3) Ansible role/playbook set for local OpenViking deployment.
4) Ansible role/playbook set for k3s/k8s OpenViking deployment.
5) Agent routing and context-backend configuration for Hermes/OpenClaw/Noesis* agents.
6) Validation playbooks and health checks.
7) Documentation for install, update, rollback, and fallback behavior.

Acceptance criteria:
- Codex Server is installed and operational in both local and k8s/k3s variants where supported.
- OpenViking is installed and operational in both local and k8s/k3s variants where supported.
- Hermes agents can delegate appropriate dev tasks to Codex.
- Agents can use OpenViking as a context backend where configured.
- The deployment is idempotent, secure, and documented.
- Sensitive or destructive tasks are not delegated without explicit authorization.
- A fallback path exists if either backend is unavailable.
