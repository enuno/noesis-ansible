Mission:
Take the playbooks for each service in the platform and transform them into production-ready Kubernetes deployments, with a strong preference for Helm-based packaging and K3s compatibility.

Objectives:
- Convert each service playbook into a declarative Kubernetes deployment artifact.
- Prefer Helm charts for all services unless a service has a clear reason to remain as raw manifests.
- Ensure deployments are portable across Kubernetes and K3s.
- Preserve service-specific runtime requirements, dependencies, persistence, networking, and security constraints.
- Produce a clean, repeatable deployment structure suitable for GitOps, CI/CD, and environment promotion.

Scope:
- For every service playbook:
  - identify runtime image, ports, env vars, secrets, volumes, persistence, health checks, ingress, RBAC, resources, and autoscaling needs;
  - map those requirements into Kubernetes objects;
  - package the service as a Helm chart when appropriate;
  - define environment overlays for dev, staging, and production;
  - document operational runbooks for deploy, upgrade, rollback, and recovery.

Required implementation standards:
- Use Helm chart structure and templating conventions.
- Keep values configurable and avoid hardcoding environment-specific data.
- Do not place secrets directly in values files.
- Use Kubernetes Secrets or an external secret manager for sensitive values.
- Include readiness and liveness probes where applicable.
- Define resource requests and limits.
- Add service, ingress, configmap, secret, PVC, HPA, service account, and network policy resources as needed.
- Support both full Kubernetes clusters and k3s-based clusters.
- Validate charts with linting and dry-run rendering before deployment.

K3s requirements:
- Ensure charts work under K3s without relying on assumptions that only hold in managed cloud environments.
- Account for K3s Helm deployment patterns and HelmChart/HelmChartConfig compatibility where useful.
- Prefer cluster-local dependencies and simple operational assumptions.

Helm packaging requirements:
- Create a reusable chart layout for each service or a rationalized parent chart with subcharts if multiple services share patterns.
- Use semantic versioning for chart and app versions.
- Keep templates maintainable and composable.
- Use helpers and library-style reuse where appropriate.
- Include a clear values schema and environment-specific values files.
- Prepare charts for CI validation and promotion between environments.

Deliverables:
1) A Kubernetes deployment version for each service playbook.
2) Helm chart(s) for each service or service group.
3) Environment-specific values files.
4) Deployment documentation and runbooks.
5) Validation artifacts from helm lint, template rendering, and dry-run deployment checks.

Execution policy:
- Read each playbook first and extract the service contract.
- Convert the contract into Kubernetes primitives.
- Keep configuration explicit and observable.
- Prefer safe defaults, least privilege, and reversible changes.
- Escalate ambiguous or missing operational details back into the playbook rather than guessing.
- Treat deployment design as infrastructure code, not ad hoc scripting.

Acceptance criteria:
- Every service playbook has a corresponding Kubernetes deployment plan.
- Every service can be deployed via Helm or a justified raw-manifest exception.
- The result is compatible with Kubernetes and K3s.
- Secrets are externalized.
- Liveness/readiness, resource limits, and rollback guidance are included.
- The output is ready for review, testing, and CI/CD integration.

Reference guidance:
- Helm chart best practices: https://helm.sh/docs/chart_best_practices/
- K3s Helm deployment behavior: https://docs.k3s.io/add-ons/helm
