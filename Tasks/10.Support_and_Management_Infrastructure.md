Mission:
Extend the Noesis Universe agent support and management infrastructure so it can support:
- AG-UI as the agent-user interaction protocol for user-facing agent experiences.
- Agent Registry as the source of truth for discovering, governing, and deploying agents, MCP servers, skills, prompts, and related artifacts.
- Harbor as the OCI registry and artifact distribution layer for agent images, support services, and deployable infrastructure components.

Primary objective:
Use Ansible, with ~/projects/noesis-ansible/ as the source of truth, to deploy, manage, validate, and maintain the full Noesis Universe back-end infrastructure. The repo must manage agent runtimes, registries, MCP servers, support services, updates, backups, and security policy as composable infrastructure.

Architecture requirements:
1. AG-UI layer
- Treat AG-UI as the user-facing agent interaction protocol.
- Support agent backends that emit AG-UI-compatible events and accept AG-UI-compatible inputs.
- Provide deployment scaffolding for AG-UI-capable services and clients.
- Make AG-UI support explicit in the agent runtime stack where agents need human-in-the-loop interaction.

2. Agent Registry layer
- Use Agent Registry as the central catalog for agents, MCP servers, skills, and prompts.
- Register all approved deployable artifacts in the registry.
- Support discovery, versioning, promotion, and governance of agent artifacts.
- Ensure Agent Registry metadata is maintained by Ansible and kept in sync with deployed reality.
- Include registry entries for agents, tool servers, and any runtime packages that must be discoverable by NoesisPraxis.

3. Harbor layer
- Use Harbor as the OCI registry and artifact backend for images and packages.
- Store deployable agent images, worker runtimes, and related infrastructure images in Harbor.
- Support Harbor projects, repositories, access control, and lifecycle policies.
- Ensure Harbor-backed artifacts can be deployed by Ansible without manual drift.
- Use Harbor as the authoritative distribution target for build outputs when container images are part of the stack.

Operational scope:
- The Noesis Universe includes:
  - ACP registry
  - ANS
  - Agent Registry
  - A2A registry
  - MCPJungle
  - Clawvisor
  - Telegram communications
  - OpenClaw
  - Hermes
  - macOS ClawDev and HermesDev
  - Tailscale networking
  - Harbor
  - AG-UI
  - backup, restore, validation, rollback, and registry sync
- All of these must be managed through Ansible playbooks and roles.
- The repository is the source of truth for deployment, configuration, and lifecycle management.

Ansible design rules:
- Use inventories to separate local, Tailscale, and production contexts.
- Use roles for reusable deployment logic.
- Use templates for AG-UI config, registry objects, Harbor config, and service definitions.
- Use custom modules only where a reusable project-specific operation justifies it.
- Use handlers for restart/reload semantics.
- Use tags and phase-based execution.
- Use Ansible Vault for secrets after local pull from Bitwarden via `bws`.
- Keep runtime execution offline from secret backends whenever possible by relying on Vault-managed local copies.

Harbor-specific requirements:
- Create Harbor project and repository scaffolding for:
  - agent images
  - support services
  - MCP-related services
  - registry companion images
- Support Harbor authentication and project-scoped access.
- Support image versioning and promotion policies.
- Include validation for repository reachability and artifact availability.
- Keep Harbor configuration under Ansible control.

AG-UI-specific requirements:
- Make AG-UI the protocol layer for user-facing agent surfaces.
- Support event-based interaction and human-in-the-loop workflows.
- Include scaffolding for agent services that expose AG-UI-compatible endpoints.
- Include placeholders for front-end/client integration if needed, but keep this repo focused on infra and deployment.

Agent Registry-specific requirements:
- Register and manage all deployed agents, MCP servers, skills, and prompts.
- Maintain versioned metadata and deployment references.
- Keep registry state synchronized with deployed services and Harbor artifacts.
- Support governance and approval workflows.

Noesis execution rules:
- NoesisPraxis is the supervisor and orchestrator.
- Ansible is the primary deployment and maintenance engine.
- NoesisPraxis should be able to deploy and maintain the back-end stack from this repo without ad hoc manual changes.
- Any generated playbook or role must be idempotent, auditable, and safe to rerun.

Output expectations:
- When generating the repo, create:
  - README.md
  - ansible.cfg
  - requirements.yml
  - inventory/
  - playbooks/
  - roles/
  - templates/
  - files/
  - scripts/
- Include support for AG-UI, Agent Registry, Harbor, ACP, ANS, A2A, MCPJungle, Clawvisor, Telegram, OpenClaw, Hermes, macOS agents, and Tailscale.
- Preserve the local-first, reusable, composable Ansible structure.

Implementation priorities:
1. Identity and registry foundation.
2. Harbor artifact and distribution setup.
3. AG-UI user-facing interaction support.
4. MCP and agent tool access.
5. Agent runtimes and macOS deployment.
6. Networking and communications.
7. Validation, backup, restore, and rollback.
